## What it does

PayFlow lets users register a digital wallet, send money to other users in real time, and receive instant notifications when a payment arrives. Under the hood it demonstrates how a payments company might design for reliability — handling failed transfers gracefully, guaranteeing exactly-once processing, and maintaining an audit trail.

**Key user flows:**
- Register and log in with JWT + refresh token rotation
- View wallet balance and transaction history
- Transfer money to another user (with live balance update via SignalR)
- Receive a real-time notification when funds arrive
- Download a monthly spending statement (PDF generated by Azure Function)

### Services

| Service | Responsibility | Key tech |
|---|---|---|
| **Identity** | Auth, JWT, refresh tokens | ASP.NET Core Identity, Azure AD B2C, Key Vault |
| **Wallet** | Balance management, idempotent credits/debits | EF Core, CQRS/MediatR, Repository pattern |
| **Transaction** | P2P transfer lifecycle, distributed saga | MassTransit, Saga pattern, Outbox pattern |
| **Notification** | Real-time alerts, email | SignalR, Azure Service Bus, Azure Communication Services |
| **Reporting** | Spending analytics, monthly statements | Azure Functions (timer trigger), gRPC |

### Communication patterns

- **REST** — Angular SPA ↔ API Gateway ↔ services (external-facing)
- **gRPC** — Transaction service → Wallet service (balance check before debit; low latency, typed contracts)
- **Azure Service Bus** — Transaction service → Notification service (fire-and-forget, decoupled)
- **SignalR** — Notification service → Angular SPA (real-time push)

### The Outbox pattern

The Transaction service guarantees at-least-once delivery by writing both the transaction record and a Service Bus message in a single database transaction. A background `OutboxProcessor` service polls for unprocessed messages and publishes them, ensuring no event is lost even during a Service Bus outage.

```
1. BEGIN TRANSACTION
2.   INSERT INTO Transactions (status = Pending)
3.   INSERT INTO OutboxMessages (event_type, payload, processed = false)
4. COMMIT
5. [OutboxProcessor] polls WHERE processed = false → publishes to Service Bus
6. [OutboxProcessor] marks processed = true
```

## Tech stack

**Backend**
- .NET 8 Web API (Clean Architecture per service)
- MediatR — CQRS command/query separation
- Entity Framework Core — data access
- MassTransit — messaging abstraction over Azure Service Bus
- Polly — retry, circuit breaker, timeout policies
- gRPC — strongly-typed inter-service communication

**Cloud (Azure)**
- Azure App Service — hosting all .NET APIs
- Azure SQL — one database per service (no shared state)
- Azure Service Bus — async event messaging
- Azure Key Vault — secrets and connection strings (Managed Identity)
- Azure Functions — scheduled reporting jobs
- Azure Cache for Redis — distributed caching
- Application Insights — distributed tracing and performance monitoring

**Frontend**
- Angular 17 (standalone components, signals)
- NgRx — state management
- Angular Material — UI component library
- SignalR client — real-time notifications

**DevOps**
- Azure DevOps — CI/CD pipelines (YAML)
- Docker + Docker Compose — local development
- Bicep — infrastructure as code
- GitHub — source control (with branch protection + PR reviews enforced)

## Running locally

### Prerequisites
- .NET 8 SDK
- Docker Desktop
- Node.js 20+
- Azure CLI (optional, for cloud deployment)

### Steps

```bash
# Clone the repo
git clone https://github.com/your-username/payflow.git
cd payflow

# Start all backend services and databases
docker-compose up -d

# Run the Angular frontend
cd frontend/payflow-web
npm install
ng serve

# Open http://localhost:4200
```

The Docker Compose file starts all 5 services, SQL Server (local), and a local RabbitMQ instance (replaces Azure Service Bus for local dev).

## Deployment

The project deploys automatically to Azure on every merge to `main` via Azure DevOps.

**Pipeline stages:**
1. `build` — restore, build, run unit tests
2. `test` — integration tests against Docker Compose services
3. `infra` — apply Bicep templates (idempotent)
4. `deploy` — push Docker images to ACR, restart App Service slots

Infrastructure is defined in `infra/bicep/main.bicep` and can be deployed to a new Azure subscription with:

```bash
az deployment group create \
  --resource-group payflow-rg \
  --template-file infra/bicep/main.bicep \
  --parameters @infra/bicep/parameters/prod.parameters.json
```

## Key design decisions

| Decision | Choice | Reason |
|---|---|---|
| Architecture | Clean Architecture per service | Testability, separation of concerns, easy to extend |
| CQRS | MediatR | Decouples command logic from HTTP layer; enables pipeline behaviors (logging, validation) |
| Distributed transactions | Saga (choreography) | Avoids 2PC; each service owns its local transaction and compensates on failure |
| Message reliability | Outbox pattern | Guarantees at-least-once delivery without distributed transactions |
| Secrets | Azure Key Vault + Managed Identity | No credentials in code or config files |
| Inter-service calls | gRPC (internal) + REST (external) | gRPC for performance-critical paths; REST for simplicity at the API Gateway |

## What I learned building this

- Implementing the **Saga pattern** to handle a multi-step transfer that can fail at any step — and writing compensating transactions to roll back cleanly
- Why **idempotency** matters in payments: the wallet service uses an idempotency key on every debit so retries from the Transaction service never double-charge
- How the **Outbox pattern** solves the "dual write problem" that every distributed system eventually hits
- Practical **Azure Key Vault** integration using Managed Identity — no connection strings in any config file
- How **Application Insights distributed tracing** lets you follow a single payment request across all 5 services with one correlation ID
